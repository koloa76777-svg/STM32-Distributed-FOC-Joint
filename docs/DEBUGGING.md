# 工程调试与问题解决

> 返回 [README](../README.md)

以下是开发中定位并解决的若干典型问题,均来自真实调试,体现从现象到根因的排查过程。

---

## 1. 电机行为随调试打印(printf)改变 → 定位到角度更新架构缺陷

**现象**:同一套代码,主循环里加 printf 和不加 printf,电机行为完全不同——加 printf 时双向拨动才转,关掉 printf 后变成单向、且需手拨启动。这是一个极强的"架构警报":功能代码没动,仅调试语句的有无就改变控制行为,说明系统存在隐藏的时序依赖。

**根因**:当时 θe(电角度)在主循环异步更新,而 FOC 运算在 25kHz 中断里固定频率使用。printf 是阻塞式 UART 发送,它的有无直接改变主循环每圈的耗时 → θe 的更新速率随之变化 → 中断每 20μs 拿到的 θe "新鲜度"不固定 → 相位抖动、转矩方向不稳。本质是快慢环不同步:控制环用了一个更新节拍不受控的角度。

**解决**:把 θe 读取纳入控制中断节拍——SPI+DMA 在中断内非阻塞读取编码器,与电流采样同步。角度更新频率从此固定、与控制环锁定。改造后电机行为不再依赖 printf,系统从"算法正确但行为飘忽"变为确定性可控。

> 方法论:当无关代码(日志、延时)的增减会改变系统行为时,几乎一定是时序/同步问题,而非功能逻辑问题。

## 2. SPI+DMA 在高频中断中状态机死锁 → 错误自愈机制

**现象**:把 θe 读取改到中断里用 SPI+DMA 后,出现两种死锁:短期——DMA 触发过快时 HAL 报错 `HAL_SPI_ERROR_DMA`(ErrorCode=0x10),状态机卡在 BUSY;长期——运行一段时间后偶发锁死,手转电机角度也冻结不更新。

**根因**:在 25kHz 中断里"踢" DMA 太快,HAL 库的 SPI 收尾流程(状态机从 BUSY 回到 READY)跟不上。实测最小可靠重启间隔约 80μs——踢得比这更密,HAL 来不及收尾就被下一次请求打断,状态机错乱锁死。

**解决**:两手并用——

1. 分频降速:θe 读取从每拍(20μs)分频到约 10kHz(100μs),拉开到 80μs 重入边界之外。实测分频 ≥4 稳定,=3 立即卡死。
2. 错误自愈机制:每次踢 DMA 前检查 HAL State,若发现锁死(State≠READY)就 `HAL_SPI_Abort` + 强制复位状态机 + 清错误标志,下一拍自动恢复。

## 3. 单电流环空载持续加速,最终撞电源限流(CC)

**现象**:电流环单独运行、给定恒定 iq 指令,电机空载下持续加速,转速不断攀升,最终电源进入限流(CC)状态,电机走走停停。

**根因**:电流环只控制电流(转矩),不控制转速——这不是 bug,是控制结构的固有边界。恒转矩空载下转速持续上升;反电动势随转速线性增大(`E = Ke·ω`),不断吃掉母线电压裕度;当电流环输出电压饱和(顶到 `Vdc/√3`)后失去对电流的控制能力,电流畸变 + PI 积分饱和 → 撞 CC。

**解决**:这正暴露了串级控制的必要性——加外层速度环 / 位置环,给"转速"这个原本无反馈的状态量加上约束。理解这一点是从"调通电流环"到"理解三环为何存在"的关键一跃。

> 面试延伸:更高速的工业场景还会用弱磁控制(高速时给负 id 削弱反电动势),扩展调速范围。

## 4. 低速三环抖动,加滤波/加阻尼越调越糟 → 砍掉速度环,位置环 PD 直控

**现象**:标准三环(位置环套速度环)在低速下剧烈抖动,锁不稳。依次尝试:降位置环增益(好转仍抖)→ 加 D 阻尼(仍抖)→ 加死区(静止稍好、运动照抖)→ 加大测速窗口(更糟,变成等幅自激振荡)。所有"加法"都没能根治。

**根因**:旁路位置环、直接给恒定速度指令后发现,测速值 omega 本身在 ±100 范围狂跳——病根在速度测量,不在位置环。速度由编码器角度微分得到,低速时单位时间转过的角度极小(实测 1ms 内仅约 11 个编码器 LSB,而静态噪声就有 ±2~3 LSB),量化噪声占比爆炸。为压噪声被迫重滤波,而重滤波引入相位滞后、吃掉相位裕度,在闭环里超过临界点就自激振荡——陷入"不滤波则抖、滤波则振"的死循环。

**解决**:换思路做减法——位置 θ 是编码器直接测的、干净的;速度 ω 是微分出来的、脏的。让干净的位置当主控,脏的速度只做辅助阻尼:

```c
iq_ref = Kp_pos * theta_err + pos_integral - Kd_pos * omega_filt;
//       └─ 位置误差(干净)主力 ─┘  └积分补卡点┘  └速度仅阻尼(可重滤波)┘
```

位置环直接输出电流指令,绕过速度环。P/I 项吃干净的位置误差,低速不再抖;D 项的脏速度只管防过冲,滞后无所谓。低速抖动根治。

> 减法 > 加法:卡死/振荡的解法常常是删掉一个引入噪声或延迟的环节,而非再叠加补偿。

## 5. 滤波系数越小(滤波越强)反而越容易振荡

**现象**:调速度反馈的一阶低通滤波系数 α 时,出现反直觉结果——α=0.02(强滤波)剧烈振荡失稳,α=0.05 最稳,α=0.1(弱滤波)也稳但波动略大。"滤波越强反而越炸",与"滤波让信号更平滑"的直觉相悖。

**根因**:速度反馈不是拿来"看"的,它在闭环回路里。α 越小,滤波越重,omega 越平滑但滞后越严重——真实速度已经到了,滤波输出还停在旧值。速度环据此误判"还不够快"继续加电流,等滤波值追上来时真实速度早已冲过头 → 加过头→减过头→振荡发散。滤波引入的相位滞后吃掉了系统相位裕度,过度滤波越过临界点即自激振荡。

**解决**:取 α=0.05 折中。更根本的做法是从源头降噪而非末端重滤波——后续改用 M 法多点测速(拉长测速窗口分母,把量化噪声平摊降到 1/N),信号本身变干净,只需轻滤波,避免了重滤波的相位滞后。

> 同一个滤波器,开环看波形 α 越小越平滑,闭环控制 α 太小反而失稳——开环与闭环对滤波的结论相反。

## 6. 换电机后开环能转、闭环堵转锁死

**现象**:换了同型号电机(极对数、线序均未变),开环(虚拟角度)能转,但闭环堵转、上电瞬间撞 CC 锁死。

**根因**:开环能转说明算法没问题,问题在"与电机绑定的参数"。逐一排除极对数、线序后锁定编码器方向——断电手转实测,新电机"顺时针→机械角增大",与旧电机相反,而代码里 θe 的符号(`theta_offset - theta_m` 的取反)是按旧电机方向写的,方向反了 → 电压矢量对抗磁极 → 堵转撞 CC。改对符号即恢复。

> 症状组合"开环能转 + 闭环堵转 + 上电锁死 + 撞 CC",基本指向编码器方向 / θe 符号与电机没对上——换电机后这是头号排查项。

## 7. 电流采样读数冻结,疑似硬件断线

**现象**:adc_buf 数值冻结,通电断电都不变,一度怀疑采样电路硬件断了。

**根因**:用万用表实测 A0/A2 引脚电压确实随电流变化,证伪了"硬件断"的猜测——真因是 `HAL_ADC_Start` 与 `HAL_ADC_Start_DMA` 两种 API 混用,DMA 状态机错乱,buffer 不再更新。

**解决**:统一用 DMA,不混用;并确认 `DMAContinuousRequests=ENABLE`(否则 TRGO 触发后 DMA 搬一轮就停)。

> 现场观察优先于理论推断——一句"硬件可能断了"的猜测,被一次万用表实测直接推翻,省下大量瞎改时间。

## 8. SVPWM 七段式查表导致一相焊死

**现象**:用经典"判扇区 + X/Y/Z 查表"方案,某一相 PWM 焊死(钉在 1020+ 不动)、电流偏小、电机抖动。

**根因**:查表的扇区映射表来自与本实现不同的扇区编号约定,映射错位导致该相输出错误。

**解决**:改用数学等价的 min-max 共模注入法(见 [TECHNICAL.md](./TECHNICAL.md)),无扇区查表、只需比较加减,一相焊死问题消失,三相对称平稳。

> 当一个方案的正确性依赖"一张容易抄错的表",换用数学等价但无表的实现往往更鲁棒。
